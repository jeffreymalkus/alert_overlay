Key finding: 7 of 12 strategies use a $0.02 fixed buffer with only 0.15 ATR minimum floor. On a $50 stock with 5-min ATR of $0.30, the minimum stop is $0.045 — that's 0.09% of price. At 16bps round-trip friction, cost/R = 0.0016 × $50 / $0.045 = 1.78R. The trade is dead before it starts.

The short strategies (BDR, ORH, PDH) use ATR-based buffers with wider floors (0.30–0.50 ATR), giving much better cost economics — but still not enough.

Q2. How is target price chosen?

Every strategy uses compute_structural_target_long() or _short() with this logic:

  1. Collect candidate structural levels (VWAP, session high/low, PDH/PDL, measured moves, etc.)
  2. Filter: keep only candidates that are >= min_rr away from entry (all strategies: min_rr = 1.0)
  3. Cap: clamp each candidate at max_rr (all strategies: max_rr = 3.0)
  4. Select: pick the nearest viable candidate (smallest R:R that meets min_rr)
  5. Fallback: if no structural candidate qualifies, use fallback_rr as a fixed target

All targets are structural or structural-fallback. No strategy forces a fixed R:R anymore. The actual_rr stored in metadata reflects the real structural distance (1.0 to 3.0 range). The fallback_rr only kicks in when no structural candidate meets min_rr = 1.0.

Q3. What does "Target RR" in the economics table represent?

The tgtRR column in the economics table shows the median actual_rr across all trades for that strategy — i.e., the median structural target distance selected by compute_structural_target.

This is NOT the fixed configured fallback. It is the real distance to the structural level that was chosen for each trade, capped at max_rr=3.0 and floored at min_rr=1.0.

The STRATEGY_TARGET_RR dict in replay_live_path.py stores the fallback values used when actual_rr is missing from metadata. But for most trades, the metadata actual_rr is present and reflects the real structural target.

Q4. When target is hit in simulation, how is it filled?

Exact touch = full fill. No buffer. No confirmation.

The code (helpers.py lines 301-313):
  • LONG: if b.high >= target → fill at target price, pnl = target_rr - cost_rr
  • SHORT: if b.low <= target → fill at target price, pnl = target_rr - cost_rr
This means:
  • If the bar's high just touches the target and reverses, the sim counts it as a full fill
  • No "trade through" requirement — a wick to the level = filled
  • No slippage buffer on the target fill itself
  • The exit fill price is exactly target_price, never worse
This overstates target hit rate. In live trading, limit orders at a level only fill if price trades through. Some percentage of these "target hits" would become EOD or stop exits in reality.

Q5. When no target/stop is hit, how is exit handled?

Three paths to exit without target/stop:

A. EOD exit (time of day)

if hhmm >= 1555 → exit at bar's close price.
P&L = (close - entry) × direction / risk - cost_rr.
This is the primary time-based exit. Realistic.

B. Max bars fallthrough

If the loop exhausts max_bars iterations without hitting EOD/stop/target, exit at the last bar's close price.
P&L = (last_bar.close - entry) × direction / risk - cost_rr.

This was fixed in the current session. Previously, the fallthrough assigned pnl_rr = 0.0 — a free pass that masked real losses. Now it computes actual P&L from the close price.
